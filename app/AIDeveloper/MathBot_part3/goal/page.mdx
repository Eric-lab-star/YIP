### 📬 글자 핸들러 vs 사진 핸들러

지금까지 만든 봇은 **글자 메시지**만 받아서 답장했다냥. 그런데 사진은 글자와 완전히 다른 종류의 데이터라냥. 그래서 사진을 받아 처리하는 **사진 전용 일꾼(사진 핸들러, PhotoHandler)**을 따로 만들어줘야 한다냥!

### 🔄 사진 한 장이 풀이로 바뀌는 과정

<StepCard step="흐름" title="사진을 받으면 이런 순서로 처리된다냥!">

1. **사진 받기:** 사용자가 수학 문제 사진을 봇에게 보낸다냥.
2. **파일로 저장:** 받은 사진을 컴퓨터에 파일(`received.jpg`)로 저장한다냥.
3. **풀이 생성:** 저장한 사진을 17차시에서 만든 풀이 기능(AI)에 전달한다냥.
4. **답장 전송:** AI가 만든 풀이 결과를 사용자에게 답장으로 보낸다냥.

</StepCard>

<Callout type="tip">
  💡 **사용자를 배려하는 작은 한 줄!**

  사진을 받아 AI가 풀이를 만드는 데에는 **시간이 조금 걸린다냥.** 그동안 사용자가
  "어? 봇이 멈췄나?" 하고 불안해하지 않도록, **"잠시만 기다려주세요..."** 같은
  안내 메시지를 먼저 보내주면 훨씬 친절한 봇이 된다냥! 🐾
</Callout>

## 3단계 · 모듈식 개발이 왜 좋을까냥? (3분)

<NyangSpeech>
  우리가 기능을 처음부터 하나로 뭉쳐서 만들지 않고, **따로따로 만든 다음 마지막에
  합친** 이유가 있다냥! 이런 방식을 **모듈식 개발**이라고 부른다냥. 😼
</NyangSpeech>

<Callout type="note">
  **모듈식 개발의 장점**

  - **테스트하기 쉽다:** 각 부품을 따로 만들면, 그 부품 하나만 콕 집어서 잘 작동하는지 확인할 수 있다냥.
  - **오류를 찾기 쉽다:** 문제가 생겼을 때 "어느 부품에서 잘못됐는지" 빠르게 알 수 있다냥.
  - **다시 쓰기 좋다:** 17차시에서 만든 풀이 기능을 오늘 그대로 **재사용**하는 것처럼, 한 번 잘 만든 부품은 계속 활용할 수 있다냥!
</Callout>

## 4단계 · 풀이 프롬프트는 그대로 재사용한다냥 (기본 코드 살펴보기)

<NyangSpeech>
  오늘 봇의 머리(AI)에게 줄 명령서, 즉 **풀이 프롬프트**는 17차시에서 만든 걸
  그대로 가져다 쓴다냥. 새로 만들 필요 없이 부품을 재사용하는 거다냥! 🐱
</NyangSpeech>

```python
# 17차시에서 만든 풀이 프롬프트 재사용
solve_prompt = """
너는 친절한 수학 선생님이야. 이 사진 속 수학 문제를 풀어줘.
문제를 설명하고, 단계별 풀이 과정을 보여준 뒤, 마지막에 '정답: '으로 정답을 알려줘.
중학생도 이해할 수 있게 쉽게 설명해줘.
"""
```

이 프롬프트는 AI에게 **"어떻게 풀이를 만들어야 하는지"** 알려주는 안내문이다냥. 한 번 잘 만들어두면, 사진만 바뀌어도 똑같이 친절한 풀이를 만들어준다냥! 😺
